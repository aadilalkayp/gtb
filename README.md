# GTB OS

Operating system for **Groom To Be** / **Glow To Be** wedding transformation programs.

Manages the full client lifecycle: leads, onboarding, plans, payments, consultant
assignments, consultations, styling operations, follow-ups, expenses, and reporting —
plus a client-facing portal.

## Stack

| Layer                | Tech                                         |
| -------------------- | -------------------------------------------- |
| Frontend             | Vite + React (SPA) → Cloudflare Pages        |
| Backend              | Next.js API routes → VPS                     |
| Database             | Supabase (Postgres)                          |
| ORM / Access control | ZenStack (on top of Prisma)                  |
| Data fetching        | TanStack Query (hooks generated by ZenStack) |
| Auth                 | Supabase Auth                                |
| Storage              | Supabase Storage                             |
| Email                | Mailgun (SMTP)                               |

## Monorepo layout

```
apps/
  web/        Vite + React SPA (staff portal + client portal)
  api/        Next.js backend (ZenStack server adapter, business logic)
packages/
  db/         ZenStack schema, generated Prisma client + TanStack hooks
  shared/     Shared constants, enums, types, validators
```

## Getting started

```bash
pnpm install

# 1. Create a Supabase project, copy credentials into the .env files
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env
cp packages/db/.env.example packages/db/.env

# 2. Generate the Prisma client + hooks from the ZenStack schema
pnpm db:generate

# 3. Push the schema to your database
pnpm db:migrate

# 4. Seed the founder account, lead sources, and example plans
pnpm --filter @gtb/db seed

# 5. Run everything
pnpm dev
```

### Storage & email

- Create a **private** Supabase Storage bucket named `client-documents`. The
  onboarding flow uploads payment proofs and photos there via the API's
  service-role key (no bucket RLS needed).
- Mailgun SMTP (`MAILGUN_SMTP_*` in `apps/api/.env`) powers invitation emails. If
  it's unset, invites still work — the registration link is returned to the staff
  UI to copy and share manually.

See [`gtb_os_srs_v2.md`](./gtb_os_srs_v2.md) for the full specification.

## Required environment variables

See each app's `.env.example`. At minimum you'll need a Supabase project URL +
keys, a Postgres connection string, and Mailgun SMTP credentials.
